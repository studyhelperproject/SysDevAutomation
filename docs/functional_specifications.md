# 機能仕様書 (Functional Specifications)

## 1. システムの目的
本システムは、顧客からの曖昧な要望をAI（Gemini）が解析し、エンジニアが即座に開発に着手できる形式（GitHub Issue）に自動変換することを目的とします。

## 2. 主要機能

### 2.1 Slack-GitHub 連携
- **自動リポジトリ管理**:
  - Slackチャンネルにボットが追加された際、または初めてメッセージを受信した際、そのチャンネル専用のGitHubリポジトリを自動的に作成（または既存のものを紐付け）します。
  - リポジトリ名は `slack-<channel_name>-<channel_id>` の形式で生成されます。
- **入力インターフェース**:
  - ボットへのメンション、またはボットとのダイレクトメッセージ（IM）を介して要望を受け付けます。
- **双方向のコンテキスト保持**:
  - 要望の解析時、GitHub上の既存Issue（オープンなもの）を読み込み、Geminiにコンテキストとして提供します。
  - Slackのスレッド履歴（最新20件）を解析に含め、会話の流れを考慮した応答を行います。
- **トレーサビリティ**:
  - 作成・更新されるGitHub Issueには、元となったSlackメッセージへのリンク（パーマリンク）が自動的に付与されます。

### 2.2 要件の分類と構造化
Geminiは入力を以下のカテゴリに分類し、GitHub Issueのラベルとして付与します。

| カテゴリ | 内容 | GitHubラベル |
| :--- | :--- | :--- |
| **[Feature]** | 実装すべき明確な機能 | `[Feature]` |
| **[Clarify]** | 定義不足で、詳細化が必要な項目 | `[Q]` |
| **[Dependency]** | 素材提供や外部依存が必要な項目 | `[Dependency]` |
| **[Estimate]** | 工数増加が伴う追加要望 | `[Estimate]` |
| **[Out of Scope]** | 今回のスコープ外（将来的な要望） | `[Out of Scope]` |

### 2.3 曖昧さの自動検知 (Ambiguity Detection)
- 「いい感じに」「適宜」「お任せ」といった曖昧な表現や、具体的な数値・仕様が欠けている場合、AIは自動的に `is_ambiguous: true` と判定します。
- 曖昧と判定された場合、カテゴリは強制的に `[Clarify]` (ラベル: `[Q]`) となり、解決のための「逆質問（Missing Information）」が生成されます。

### 2.4 受入基準 (Acceptance Criteria) の生成
- `[Feature]` と判定された要望に対しては、エンジニアがテスト可能な形式である「Given/When/Then」形式で受入基準を自動生成します。

### 2.5 フィードバックループ
- **Slackへの即時応答**:
  - GitHubでのアクション（Issue作成、更新、コメント追加）の結果をSlackに返信します。
  - `[Clarify]` や `[Dependency]` の場合、ユーザーが回答すべき具体的な質問や要求事項をSlack上に表示します。
- **情報の集約**:
  - 曖昧な点についてユーザーが追加情報を送ると、既存の `[Clarify]` Issueが自動的に更新、またはコメントが追加され、情報が1つのIssueに集約されます。

### 2.6 ストーリーポイント (SP) の集計
- GitHub Issueに付与された `SP: <number>` ラベルを集計し、プロジェクト全体の進捗状況をREADMEに自動反映します。
